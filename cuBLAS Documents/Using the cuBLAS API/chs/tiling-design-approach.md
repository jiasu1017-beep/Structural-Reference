### 4.1.1. 瓦片化设计方案

为了能够在多个 GPU 之间共享工作负载，cuBLASXt API 采用了一种瓦片化策略：将每个矩阵划分为用户可控尺寸为 `BlockDim x BlockDim` 的方形瓦片。生成的矩阵瓦片化定义了静态调度策略：每个生成的瓦片以轮询方式分配给一个 GPU。为每个 GPU 创建一个 CPU 线程，负责执行适当的内存传输和 cuBLAS 操作，以计算其负责的所有瓦片。从性能角度来看，由于这种静态调度策略，最好让每个 GPU 的计算能力和 PCI 带宽保持一致。下图说明了 3 个 GPU 之间的瓦片分布。为了计算 C 中的第一个瓦片 G0，负责 GPU0 的 CPU 线程 0 必须以流水线方式从 A 的第一行加载 3 个瓦片，并从 B 的第一列加载瓦片，以重叠内存传输和计算，并将结果求和到 C 的第一个瓦片 G0 中，然后继续处理下一个瓦片 G0。

图 3. `cublasXt<t>gemm()` 在 3 个 GPU 上的瓦片化示例

当瓦片维度不是 C 维度的精确倍数时，某些瓦片在右侧边界和/或底部边界会被部分填充。当前实现不会对这些不完整的瓦片进行填充，而是通过执行正确的简化 cuBLAS 操作来跟踪这些不完整的瓦片：这样就不会执行额外的计算。然而，当所有 GPU 需要处理的不完整瓦片数量不同时，仍然可能导致负载不平衡。

当一个或多个矩阵位于某些 GPU 设备上时，会应用相同的瓦片化方法和负载共享。内存传输在这种情况下在设备之间进行。但是，当某个瓦片的计算和某些数据位于同一个 GPU 设备上时，本地数据的内存传输会被绕过，GPU 直接在本地数据上操作。这可以带来显著的性能提升，特别是在仅使用一个 GPU 进行计算时。

矩阵可以位于任何 GPU 设备上，不必位于同一个 GPU 设备上。此外，矩阵甚至可以位于不参与计算的 GPU 设备上。

与 cuBLAS API 相反，即使所有矩阵都位于同一设备上，cuBLASXt API 从主机角度来看仍然是阻塞 API：无论数据位于何处，返回调用时数据结果都是有效的，无需进行设备同步。
