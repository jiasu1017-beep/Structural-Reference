## 4.1. 概述

cuBLAS 的 cuBLASXt API 提供了一个支持多 GPU 的主机端接口：使用此 API 时，应用程序只需在主机内存空间中分配所需的矩阵。此外，当前实现在支持计算能力 6.x 或更高版本的 GPU 设备的 Linux 系统上支持托管内存，但将其视为主机内存。Windows 系统不支持托管内存。对矩阵大小没有限制，只要它们能够装入主机内存即可。cuBLASXt API 负责在指定的 GPU 之间分配内存并在它们之间调度工作负载，最后将结果检索回主机。cuBLASXt API 仅支持计算密集型的 BLAS3 例程（例如矩阵-矩阵运算），在这些运算中，往返 GPU 的 PCI 传输可以被摊销。cuBLASXt API 有其自己的头文件 `cublasXt.h`。

从 8.0 版本开始，cuBLASXt API 允许将任何矩阵放置在 GPU 设备上。

### 4.1.1. 分块设计方法

为了在多个 GPU 之间共享工作负载，cuBLASXt API 采用分块策略：将每个矩阵划分为用户可控尺寸 BlockDim x BlockDim 的方形块。由此产生的矩阵分块定义了静态调度策略：每个结果块以轮询方式分配给一个 GPU。为每个 GPU 创建一个 CPU 线程，负责执行适当的内存传输和 cuBLAS 操作来计算其负责的所有块。从性能角度来看，由于这种静态调度策略，最好让每个 GPU 具有相同的计算能力和 PCI 带宽。下图说明了 3 个 GPU 之间的块分布。要计算 C 中第一个块 G0，负责 GPU0 的 CPU 线程 0 必须以流水线方式从 A 的第一行加载 3 个块，并从 B 的第一列加载块，以重叠内存传输和计算，并在移动到下一个 G0 块之前将结果求和到 C 的第一个 G0 块中。

cublasXt<t>gemm() 在 3 个 GPU 上的分块示例

当块尺寸不是 C 尺寸的精确倍数时，某些块在右边框或/和下边框处部分填充。当前实现不会填充不完整的块，而是通过执行正确的简化 cuBLAS 操作来跟踪这些不完整的块：这样就不会执行额外的计算。但是，当所有 GPU 处理的不完整块数量不同时，仍可能导致负载不平衡。

当一个或多个矩阵位于某些 GPU 设备上时，会应用相同的分块方法和工作负载共享。在这种情况下，内存传输在设备之间完成。但是，当某个块的计算和某些数据位于同一 GPU 设备上时，到/从本地数据的内存传输会被绕过，GPU 直接在本地数据上操作。这可以带来显著的性能提升，尤其是在仅使用一个 GPU 进行计算时。

矩阵可以位于任何 GPU 设备上，不必位于同一 GPU 设备上。此外，矩阵甚至可以位于不参与计算的 GPU 设备上。

与 cuBLAS API 相反，即使所有矩阵位于同一设备上，从主机角度来看，cuBLASXt API 仍然是阻塞 API：无论结果位于何处，都将在调用返回时有效，不需要设备同步。

### 4.1.2. 混合 CPU-GPU 计算

对于非常大的问题，cuBLASXt API 提供了将部分计算卸载到主机 CPU 的可能性。可以使用例程 cublasXtSetCpuRoutine() 和 cublasXtSetCpuRatio() 设置此功能。分配给 CPU 的工作负载被搁置一旁：这只是从底部和右侧取出的结果矩阵的百分比，取尺寸较大的那一侧。GPU 分块在缩减后的结果矩阵上进行。

如果任何矩阵位于 GPU 设备上，则此功能将被忽略，所有计算将仅在 GPU 上完成。

应谨慎使用此功能，因为它可能会干扰负责向 GPU 提供数据的 CPU 线程。

目前，只有例程 cublasXt<t>gemm() 支持此功能。

### 4.1.3. 结果可重现性

当前，在满足以下条件时，来自给定工具包版本的所有 cuBLASXt API 例程会产生相同的位级结果：

- 参与计算的所有 GPU 具有相同的计算能力和相同的 SM 数量。
- 块大小在各次运行之间保持相同。
- 要么未使用 CPU 混合计算，要么提供的 CPU BLAS 也能保证产生可重现的结果。
