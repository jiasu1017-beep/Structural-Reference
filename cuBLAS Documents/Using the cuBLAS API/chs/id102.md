## 4.1. 概述

cuBLAS 的 cuBLASXt API 公开了一个支持多 GPU 的主机端接口：使用此 API 时，应用程序只需要在主机内存空间中分配所需的矩阵。此外，当前实现在具有计算能力 6.x 或更高版本的 GPU 设备上支持 Linux 上的托管内存，但将其作为主机内存处理。托管内存在 Windows 上不受支持。只要矩阵能够容纳在主机内存中，对其大小没有限制。cuBLASXt API 负责在指定的 GPU 之间分配内存，在它们之间分派工作负载，最后将结果检索回主机。cuBLASXt API 仅支持计算密集型的 BLAS3 例程（例如矩阵-矩阵运算），其中可以摊销 GPU 往返的 PCI 传输。cuBLASXt API 有自己的头文件 `cublasXt.h`。

从 8.0 版本开始，cuBLASXt API 允许任意矩阵位于 GPU 设备上。

>
> 注意
> 使用流式顺序内存分配器分配在 GPU 上的矩阵时，请使用 `cudaMemPoolSetAccess` 确保所有设备之间的可见性。

> **注意**
>
> 注意
> cuBLASXt API 仅在 64 位平台上受支持。

### 4.1.1. 分块设计方法

为了能够在多个 GPU 之间共享工作负载，cuBLASXt API 使用分块策略：将每个矩阵划分为用户可控尺寸 BlockDim × BlockDim 的方形块。生成的矩阵分块定义了静态调度策略：每个生成的块以循环方式分配给 GPU。为每个 GPU 创建一个 CPU 线程，负责执行适当的内存传输和 cuBLAS 操作，以计算其负责的所有块。从性能角度来看，由于这种静态调度策略，最好让每个 GPU 具有相同的计算能力和 PCI 带宽。下图说明了这 3 个 GPU 之间的块分布。为了从 C 计算第一个块 G0，负责 GPU0 的 CPU 线程 0 必须以流水线方式从 A 的第一行加载 3 个块，从 B 的第一列加载块，以重叠内存传输和计算，并在移动到下一个块 G0 之前将结果求和到 C 的第一个块 G0 中。

cublasXt<t>gemm() 在 3 个 GPU 上的分块示例

当块尺寸不是 C 尺寸的整数倍时，某些块在右边界或/和底边界被部分填充。当前实现不会填充不完整的块，而是通过执行正确的简化 cuBLAS 操作来跟踪这些不完整的块：这样就不会进行额外的计算。但是，当所有 GPU 处理的不完整块数量不同时，仍然可能导致负载不平衡。

当一个或多个矩阵位于某些 GPU 设备上时，将应用相同的分块方法和工作负载共享。在这种情况下，内存传输在设备之间进行。但是，当某个块的计算和某些数据位于同一 GPU 设备上时，将绕过与本地数据之间的内存传输，GPU 直接在本地数据上操作。这可以带来显著的性能提升，特别是当仅使用一个 GPU 进行计算时。

矩阵可以位于任何 GPU 设备上，不要求位于同一 GPU 设备上。此外，矩阵甚至可以位于不参与计算的 GPU 设备上。

与 cuBLAS API 相反，即使所有矩阵位于同一设备上，cuBLASXt API 从主机角度来看仍然是阻塞 API：无论位于何处，数据结果在调用返回时都将有效，无需设备同步。

### 4.1.2. 混合 CPU-GPU 计算

对于非常大的问题，cuBLASXt API 提供了将部分计算卸载到主机 CPU 的可能性。可以使用例程 `cublasXtSetCpuRoutine()` 和 `cublasXtSetCpuRatio()` 设置此功能。分配给 CPU 的工作负载被搁置：它只是从结果矩阵的底部和右侧取出的百分比，取尺寸较大的那一侧。在缩减后的结果矩阵上完成 GPU 分块。

如果任意矩阵位于 GPU 设备上，则此功能将被忽略，所有计算都将在 GPU 上完成。

应谨慎使用此功能，因为它可能会干扰负责为 GPU 提供数据的 CPU 线程。

目前，只有例程 `cublasXt<t>gemm()` 支持此功能。

### 4.1.3. 结果可重现性

当前，来自同一工具包版本的所有 cuBLASXt API 例程在满足以下条件时会产生相同的按位结果：

- 所有参与计算的 GPU 具有相同的计算能力和相同的 SM 数量。
- 两次运行之间保持相同的块大小。
- 要么未使用 CPU 混合计算，要么提供的 CPU Blas 也能保证产生可重现的结果。
