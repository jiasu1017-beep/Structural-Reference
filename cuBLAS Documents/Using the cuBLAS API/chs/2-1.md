## 2.1. 总体描述

本节介绍如何使用 cuBLAS 库 API。

### 2.1.1. 错误状态

所有 cuBLAS 库函数调用都会返回错误状态 cublasStatus_t。

### 2.1.2. cuBLAS 上下文

应用程序必须通过调用 cublasCreate() 函数来初始化 cuBLAS 库上下文的句柄。然后，该句柄会被显式传递给每个后续的库函数调用。一旦应用程序完成使用该库，它必须调用 cublasDestroy() 函数来释放与 cuBLAS 库上下文相关的资源。

这种方法允许用户在多个主机线程和多个 GPU 上使用时显式控制库设置。例如，应用程序可以使用 `cudaSetDevice()` 将不同的设备关联到不同的主机线程，并且在每个主机线程中初始化一个唯一的 cuBLAS 库上下文句柄，该句柄将使用与该主机线程关联的特定设备。然后，使用不同句柄进行的 cuBLAS 库函数调用将自动将计算分发到不同的设备。

与特定 cuBLAS 上下文关联的设备在相应的 cublasCreate() 和 cublasDestroy() 调用之间应保持不变。为了让 cuBLAS 库在同一主机线程中使用不同的设备，应用程序必须通过调用 `cudaSetDevice()` 来设置要使用的新设备，然后通过调用 cublasCreate() 创建另一个将与新设备关联的 cuBLAS 上下文。当有多个设备可用时，应用程序必须确保在使用该上下文调用 cuBLAS 函数之前，与给定 cuBLAS 上下文关联的设备是当前的（例如通过调用 `cudaSetDevice()`）。

cuBLAS 库上下文与 cublasCreate() 调用时当前的 CUDA 上下文紧密耦合。使用多个 CUDA 上下文的应用程序需要为每个 CUDA 上下文创建一个 cuBLAS 上下文，并确保前者永远不会比后者存在更长时间。从版本 12.8 开始，cuBLAS 会检测底层 CUDA 上下文是否与图形上下文绑定，并遵循在这种情况下设置的共享内存大小限制。

### 2.1.3. 线程安全

该库是线程安全的，其函数可以从多个主机线程调用，即使使用相同的句柄也可以。当多个线程共享同一个句柄时，在更改句柄配置时需要格外小心，因为该更改可能会影响所有线程中潜在的后续 cuBLAS 调用。对于句柄的销毁更是如此。因此，不建议多个线程共享同一个 cuBLAS 句柄。

当同一句柄与用户提供的工作空间一起从多个线程使用时，需要额外的注意事项。详情请参见 cublasSetWorkspace()。

### 2.1.4. 结果可重现性

根据设计，来自特定工具包版本的所有 cuBLAS API 例程，在具有相同架构和相同数量流多处理器的 GPU 上每次运行时都会生成相同的位级结果。但是，不能保证跨工具包版本的位级可重现性，因为实现可能会因某些实现变化而有所不同。

当多个 CUDA 流处于活动状态或使用定点仿真时，此保证不再成立。如果有多个并发流处于活动状态，库可能会通过选择不同的内部实现来优化整体性能。

> **注意**
>
> 注意
> 多流执行的非确定性行为是由于库在选择并行流中运行的例程的内部工作空间时进行了优化。为了避免这种效果，用户可以：
>
> - 使用 cublasSetWorkspace() 函数为每个使用的流提供独立的工作空间，或
> - 每个流对应一个 cuBLAS 句柄，或
> - 使用 cublasLtMatmul() 而不是 GEMM 系列函数，并提供用户拥有的工作空间，或
> - 将调试环境变量 CUBLAS_WORKSPACE_CONFIG 设置为 :16:8（可能会限制整体性能）或 :4096:8（将增加库在 GPU 内存中的占用约 24MiB）。

定点仿真的非确定性行为是由于大型工作空间内存需求造成的（详见定点工作空间要求）。这需要使用 cudaMallocAsync() 动态分配内存，分配失败会导致回退到非仿真例程。为了避免这种效果，用户可以通过 cublasSetWorkspace() 提供工作空间，以满足定点仿真工作空间要求。

任何这些设置都将允许确定性行为，即使多个并发流共享一个 cuBLAS 句柄。

此行为预计在未来的版本中会有所改变。

对于某些例程，如 cublas<t>symv() 和 cublas<t>hemv()，可以使用例程 cublasSetAtomicsMode() 选择一个显著更快的替代例程。在这种情况下，由于计算中使用了原子操作，结果不能保证是位级可重现的。

### 2.1.5. 标量参数

使用标量参数的函数分为两类：

- 函数接受在主机或设备上通过引用传递的 alpha 和/或 beta 参数作为缩放因子，例如 gemm。
- 函数在主机或设备上返回标量结果，例如 amax()、amin()、asum()、rotg()、rotmg()、dot() 和 nrm2()。

对于第一类函数，当指针模式设置为 `CUBLAS_POINTER_MODE_HOST` 时，标量参数 `alpha` 和/或 `beta` 可以在堆栈上或堆上分配，不应放在托管内存中。在底层，与这些函数相关的 CUDA 内核将使用 `alpha` 和/或 `beta` 的值启动。因此，如果它们是在堆上分配的，即使内核启动是异步的，也可以在调用返回后立即释放它们。当指针模式设置为 `CUBLAS_POINTER_MODE_DEVICE` 时，`alpha` 和/或 `beta` 必须在设备上可访问，并且直到内核完成前不应修改其值。请注意，由于 `cudaFree()` 执行隐式的 `cudaDeviceSynchronize()`，因此仍然可以在调用后立即对 `alpha` 和/或 `beta` 调用 `cudaFree()`，但在这种情况下会违背使用此指针模式的目的。

对于第二类函数，当指针模式设置为 `CUBLAS_POINTER_MODE_HOST` 时，这些函数会阻塞 CPU，直到 GPU 完成其计算并将结果复制回主机。当指针模式设置为 `CUBLAS_POINTER_MODE_DEVICE` 时，这些函数会立即返回。在这种情况下，与矩阵和向量结果类似，标量结果仅在 GPU 上完成例程执行后才准备就绪。这需要适当的同步才能从主机读取结果。

无论哪种情况，指针模式 `CUBLAS_POINTER_MODE_DEVICE` 允许库函数与主机完全异步执行，即使 `alpha` 和/或 `beta` 是由先前的内核生成的。例如，当使用 cuBLAS 库实现线性系统和特征值问题的迭代求解方法时，可能会出现这种情况。

### 2.1.6. 使用流实现并行性

如果应用程序使用多个独立任务计算的结果，可以使用 CUDA™ 流来重叠这些任务中执行的计算。

应用程序可以在概念上，将每个流与每个任务相关联。为了实现任务之间的计算重叠，用户应该使用 `cudaStreamCreate()` 函数创建 CUDA™ 流，并在调用实际 cuBLAS 例程之前通过调用 cublasSetStream() 来设置每个单独的 cuBLAS 库例程要使用的流。请注意，cublasSetStream() 会将用户提供的工作空间重置为默认工作空间池；请参阅 cublasSetWorkspace()。然后，在 GPU 上可能的情况下，在单独流中执行的计算将自动重叠。当单个任务执行的计算相对较小且不足以使 GPU 保持满负荷时，这种方法特别有用。

我们建议使用新的 cuBLAS API，将标量参数和结果通过设备内存中的引用传递，以在使用流时实现最大程度的计算重叠。

流的特定应用，即多个小内核的批处理，将在下一节中描述。

### 2.1.7. 内核批处理

在本节中，我们将解释如何使用流来批处理小内核的执行。例如，假设我们有一个应用程序，需要对密集矩阵进行许多小的独立矩阵-矩阵乘法运算。

很明显，即使有数百万个小的独立矩阵，我们也无法达到与单个大矩阵相同的 *GFLOPS* 速率。例如，单个 $n \times n$ 大矩阵-矩阵乘法对 $n^{2}$ 输入大小执行 $n^{3}$ 次运算，而 1024 个 $\frac{n}{32} \times \frac{n}{32}$ 小矩阵-矩阵乘法对相同的输入大小执行 $1024\left( \frac{n}{32} \right)^{3} = \frac{n^{3}}{32}$ 次运算。然而，显而易见的是，与单个小型矩阵相比，我们可以通过许多小的独立矩阵实现显著更好的性能。

GPU 架构系列允许我们同时执行多个内核。因此，为了批处理独立内核的执行，我们可以在单独的流中运行每个内核。特别是在上面的示例中，我们可以使用 `cudaStreamCreate()` 函数创建 1024 个 CUDA™ 流，然后在每次调用 cublas<t>gemm() 之前调用 cublasSetStream()，为每个矩阵-矩阵乘法设置不同的流（请注意，cublasSetStream() 会将用户提供的工作空间重置为默认工作空间池，请参阅 cublasSetWorkspace()）。这将确保在可能的情况下，不同的计算将并发执行。虽然用户可以创建许多流，但实际上不可能有超过 32 个并发内核同时执行。

### 2.1.8. 缓存配置

在某些设备上，L1 缓存和共享内存使用相同的硬件资源。缓存配置可以使用 CUDA Runtime 函数 cudaDeviceSetCacheConfig 直接设置。缓存配置也可以使用例程 cudaFuncSetCacheConfig 针对某些函数进行特定设置。有关缓存配置设置的详细信息，请参阅 CUDA Runtime API 文档。

由于从一个配置切换到另一个配置可能会影响内核并发性，cuBLAS 库不设置任何缓存配置偏好，而是依赖当前设置。但是，某些 cuBLAS 例程（尤其是 Level-3 例程）严重依赖共享内存。因此，缓存偏好设置可能会对其性能产生不利影响。

### 2.1.9. 静态库支持

cuBLAS 库也以静态形式提供，在 Linux 上为 `libcublas_static.a`。静态 cuBLAS 库和所有其他静态数学库依赖于一个名为 `libculibos.a` 的通用线程抽象层库。

例如，在 Linux 上，要编译一个使用 cuBLAS 的小应用程序（链接到动态库），可以使用以下命令：

```c++
nvcc myCublasApp.c  -lcublas  -o myCublasApp
```

而要链接到静态 cuBLAS 库，必须使用以下命令：

```c++
nvcc myCublasApp.c  -lcublas_static   -lculibos -o myCublasApp
```

也可以使用原生主机 C++ 编译器。根据主机操作系统的不同，链接行中可能需要一些额外的库，如 `pthread` 或 `dl`。建议在 Linux 上使用以下命令：

```c++
g++ myCublasApp.c  -lcublas_static   -lculibos -lcudart_static -lpthread -ldl -I <cuda-toolkit-path>/include -L <cuda-toolkit-path>/lib64 -o myCublasApp
```

请注意，在后一种情况下，不需要 `cuda` 库。CUDA Runtime 会在需要时尝试显式打开 `cuda` 库。对于未安装 CUDA 驱动程序的系统，这允许应用程序优雅地处理此问题，并在有仅 CPU 路径可用时可能运行。

从 11.2 版本开始，使用类型化函数而不是扩展函数（cublas**Ex()）有助于在链接到静态 cuBLAS 库时减小二进制文件大小。

### 2.1.10. GEMM 算法数值行为

某些 GEMM 算法沿 K 维度拆分计算以提高 GPU 利用率，尤其是在 K 维度与 M 和 N 维度相比很大时。当 cuBLAS 启发式或用户显式选择这种类型的算法时，每个拆分的结果会被确定性求和到结果矩阵中，以获得最终结果。

对于例程 cublas<t>gemmEx() 和 cublasGemmEx()，当计算类型大于输出类型时，拆分块的和可能会导致某些中间溢出，从而产生具有一些溢出的最终结果矩阵。如果所有点积在最终转换为输出类型之前以计算类型累积，则这些溢出可能不会发生。当 computeType 为 `CUDA_R_32F` 且 Atype、Btype 和 Ctype 为 `CUDA_R_16F` 时，这种计算副作用很容易暴露。可以使用 cublasSetMathMode() 通过计算精度模式 `CUBLAS_MATH_DISALLOW_REDUCED_PRECISION_REDUCTION` 来控制此行为。

### 2.1.11. Tensor Core 使用

Tensor Core 首次随 Volta GPU（计算能力 7.0 及以上）引入，显著加速了矩阵乘法。从 cuBLAS 版本 11.0.0 开始，库可能会在可能的情况下自动利用 Tensor Core 功能，除非通过在 cuBLAS 中选择 pedantic 计算模式来显式禁用它们（请参阅 cublasSetMathMode()、cublasMath_t）。

应该注意的是，库会在它确定能提供最佳性能的地方选择启用 Tensor Core 的实现。

使用 Tensor Core 时，要获得最佳性能，矩阵维度和指针需要满足一定的内存对齐要求。具体来说，必须满足以下所有条件才能从 Tensor Core 获得最大性能：

- ((op_A == CUBLAS_OP_N ? m : k) * AtypeSize) % 16 == 0
- ((op_B == CUBLAS_OP_N ? k : n) * BtypeSize) % 16 == 0
- (m * CtypeSize) % 16 == 0
- (lda * AtypeSize) % 16 == 0
- (ldb * BtypeSize) % 16 == 0
- (ldc * CtypeSize) % 16 == 0
- intptr_t(A) % 16 == 0
- intptr_t(B) % 16 == 0
- intptr_t(C) % 16 == 0

要使用 FP8 类型进行矩阵乘法（请参阅 8 位浮点数据类型 (FP8) 使用），必须确保矩阵维度和指针满足上面列出的最优要求。除 FP8 外，使用 Tensor Core 不再对矩阵维度和内存对齐有任何限制（从 cuBLAS 版本 11.0.0 开始）。

### 2.1.12. CUDA Graphs 支持

在大多数情况下，cuBLAS 例程可以在 CUDA Graph 流捕获中被捕获，没有任何限制。

例外情况是输出结果到主机缓冲区的例程（例如 cublas<t>dot()，同时配置了指针模式 `CUBLAS_POINTER_MODE_HOST`），因为它会强制同步。

对于输入系数（例如 `alpha`、`beta`），行为取决于指针模式设置：

- 在 CUBLAS(LT)_POINTER_MODE_HOST 的情况下，系数值被捕获到图中。
- 在使用设备指针的指针模式下，系数值在图执行时使用设备指针访问。

> **注意**
>
> 注意
> 在 CUDA Graph 流捕获中被捕获时，cuBLAS 例程可以通过使用流有序分配 API（cudaMallocAsync 和 cudaFreeAsync）来创建内存节点。但是，由于目前不支持子图或从设备启动的图中的内存节点，在此场景下捕获 cuBLAS 例程的尝试可能会失败。为避免此问题，请使用 cublasSetWorkspace() 函数来提供用户拥有的工作空间内存。

### 2.1.13. 64 位整数接口

cuBLAS 版本 12 引入了 64 位整数函数。每个 64 位整数函数等同于一个 32 位整数函数，但有以下变化：

- 函数名添加了 _64 后缀。
- 维度（问题大小）数据类型从 int 更改为 int64_t。维度示例：m、n 和 k。
- 主导维度数据类型从 int 更改为 int64_t。主导维度示例：lda、ldb 和 ldc。
- 向量增量数据类型从 int 更改为 int64_t。向量增量示例：incx 和 incy。

例如，考虑以下 32 位整数函数：

```c++
cublasStatus_t cublasSetMatrix(int rows, int cols, int elemSize, const void *A, int lda, void *B, int ldb);
cublasStatus_t cublasIsamax(cublasHandle_t handle, int n, const float *x, int incx, int *result);
cublasStatus_t cublasSsyr(cublasHandle_t handle, cublasFillMode_t uplo, int n, const float *alpha, const float *x, int incx, float *A, int lda);
```

等效的 64 位整数函数为：

```c++
cublasStatus_t cublasSetMatrix_64(int64_t rows, int64_t cols, int64_t elemSize, const void *A, int64_t lda, void *B, int64_t ldb);
cublasStatus_t cublasIsamax_64(cublasHandle_t handle, int64_t n, const float *x, int64_t incx, int64_t *result);
cublasStatus_t cublasSsyr_64(cublasHandle_t handle, cublasFillMode_t uplo, int64_t n, const float *alpha, const float *x, int64_t incx, float *A, int64_t lda);
```

并非每个函数都有 64 位整数等效函数。例如，cublasSetMathMode() 没有任何可以有意义地成为 `int64_t` 的参数。为简洁起见，64 位整数 API 不会明确列出，而只是提及它们存在于相关函数中。
