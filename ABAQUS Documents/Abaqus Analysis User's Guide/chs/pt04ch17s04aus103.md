### 概述

本节讨论使用 Abaqus 和 Dymola 进行结构对逻辑协同仿真的分析设置、执行和限制详细信息。要使用本节中描述的协同仿真，您需要 Dymola 7.2 或更高版本，并带有 `.DLL` 选项。有关更多信息，请参阅 Dassault Systmes 知识库中的"使用 Abaqus 和 Dymola 进行逻辑-物理建模的用户指南和安装指南"，网址为 [www.3ds.com/support/knowledge-base](http://www.3ds.com/support/knowledge-base)。

### 准备用于结构对逻辑协同仿真的 Abaqus 模型

要为结构对逻辑协同仿真准备 Abaqus 模型，您需要：
- 识别涉及的 Abaqus 分析步骤
- 识别要交换的接口传感器和执行器
- 定义会合方案

以下各节简要描述这些步骤。有关协同仿真分析模型创建的详细信息，请参阅 ["准备协同仿真的 Abaqus 分析," 第 17.2.1 节"](pt04ch17s02aus98.md)。

#### 识别 Abaqus 分析步骤

协同仿真相互作用的时间段定义为协同仿真事件。此事件从 Abaqus 分析步骤（协同仿真步骤）开始时开始，到该步骤结束时结束。支持协同仿真技术的任何程序类型都可以作为协同仿真步骤；但是，每个分析作业只允许一个协同仿真步骤。对于多个协同仿真步骤，您可以使用重启功能在后续分析作业中定义新的协同仿真步骤。

您可以定义一个 Abaqus 与 Dymola 在运行时交换场的协同仿真步骤。您指定定义会合方案的协同仿真控制。

| **输入文件用法：** | 在步骤定义中使用以下选项： |
| --- | --- |
| | ``` [*CO-SIMULATION](../key/key-link.md#usb-kws-hcosimulation), NAME=*co-simulation name*, PROGRAM=MULTIPHYSICS, CONTROLS=*name* [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), NAME=*name* ``` |

#### 定义传感器和执行器

您必须定义传感器和执行器。任何节点或连接器单元历史输出变量都可以定义为命名传感器。有关详细信息，请参阅 ["Abaqus/Standard 和 Abaqus/Explicit 中的传感器定义" in "输出到输出数据库," 第 4.1.3 节"](pt02ch04s01aus40.md#usb-out-odboutput-sensors)。

| **输入文件用法：** | 使用以下选项定义传感器： |
| --- | --- |
| | ``` [*OUTPUT](../key/key-link.md#usb-kws-houtput), HISTORY, FREQUENCY, SENSOR, NAME=*sensor_name* [*NODE OUTPUT](../key/key-link.md#usb-kws-hnodeoutput), NSET=*NSET_Name* *nodal output variable* ``` 使用以下选项定义执行器： ``` [*AMPLITUDE](../key/key-link.md#usb-kws-mamplitude), NAME=*name*, DEFINITION=ACTUATOR ``` 此幅值定义没有与给定模拟时间定义幅值相关的数据行。相反，给定模拟时间的幅值将从 Dymola 输出导入。用户指定的名称可以在任何可以引用幅值的 Abaqus 选项中引用（如 [*CLOAD](../key/key-link.md#usb-kws-hcload)、[*BOUNDARY](../key/key-link.md#usb-kws-hboundary)、[*FIELD](../key/key-link.md#usb-kws-hfield) 和 [*CHANGE FRICTION](../key/key-link.md#usb-kws-hchangefriction)）。例如， ``` [*CLOAD](../key/key-link.md#usb-kws-hcload), AMPLITUDE=*name* 101, 3, 1.0 ``` |

#### 识别要交换的传感器和执行器

您必须指定在协同仿真期间 Abaqus 分析中导入和/或导出的场。

导入定义确保 Dymola 输出在上次数据交换时计算的值作为 Abaqus 输入（执行器），并作为指定幅值名称的当前值给出。同样，导出定义确保 Abaqus 输出（传感器）的当前值作为数据交换时 Dymola 输入的值。

您最多可以指定 5000 个执行器/传感器。

| **输入文件用法：** | 使用以下选项识别要交换的传感器和执行器： |
| --- | --- |
| | ``` [*CO-SIMULATION REGION](../key/key-link.md#usb-kws-hcosimulationregion), TYPE=DISCRETE, IMPORT *Actuator_name1, Actuator_name2, etc.* [*CO-SIMULATION REGION](../key/key-link.md#usb-kws-hcosimulationregion), TYPE=DISCRETE, EXPORT *Sensor_name1, Sensor_name2, etc.* ``` 数据行上的用户定义名称用逗号分隔，每行八个条目。每个名称必须少于 80 个字符。如有需要，使用多行数据。 |

#### 定义会合方案：目标时间

耦合步骤是两次连续交换之间的时间段，因此定义了 Abaqus 和 Dymola 之间的交换频率。耦合步骤大小在每个耦合步骤开始时确定，用于计算目标时间（下次同步交换发生的时间）。在大多数情况下，每个 Abaqus 增量交换数据是足够的。

##### 确定目标时间

在 Abaqus 和 Dymola 之间的耦合模拟过程中，当前耦合步骤大小由两个求解器协商确定，以建立合适的耦合步骤大小。每个求解器"建议"一个耦合步骤大小。然后选择这些建议步骤大小中的较小者。Abaqus 可以建议恒定或可变的耦合步骤大小。对于可变耦合步骤大小，Abaqus 将建议的耦合步骤大小设置为自动时间积分方案计算的下一个建议增量大小。使用可变耦合步骤大小通常会获得最准确的解决方案。

Abaqus 和 Dymola 交换耦合步骤大小，并根据求解器建议值中的较小者建立"协商"耦合步骤大小。然后使用此耦合步骤大小推进下次耦合会合的目标时间。

| **输入文件用法：** | 使用以下选项让 Abaqus 建议恒定耦合步骤大小： |
| --- | --- |
| | ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), STEP SIZE=*coupling_step_size* ``` 使用以下选项让 Abaqus 建议可变耦合步骤大小： ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols)（省略 STEP SIZE 参数） ``` |

##### Abaqus 如何达到目标时间

除了定义耦合步骤大小外，您还定义协同仿真控制以描述 Abaqus 如何推进到下一个目标时间。默认情况下，Abaqus 可能在耦合步骤期间执行多个时间增量，称为"subcycling"。Subcycling 允许 Abaqus 减少当前增量大小并采用更小的增量来求解非线性事件。

或者，您可以强制 Abaqus 使用等于协商耦合步骤大小的增量大小，称为"lockstep"。当以 lockstep 方式推进时，Abaqus 将无法减小时间增量来求解非线性事件。因此，在非线性事件要求减小增量大小的情况下，Abaqus 将终止模拟。

| **输入文件用法：** | 使用以下选项让 Abaqus 建议恒定耦合步骤大小： |
| --- | --- |
| | ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), TIME INCREMENTATION=LOCKSTEP ``` 使用以下选项让 Abaqus 建议可变耦合步骤大小： ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), TIME INCREMENTATION=SUBCYCLE ``` |

##### 达到目标时间

您可以控制 Abaqus 达到目标时间的准确程度。Abaqus 目标时间可以"精确"或"宽松"地达到。默认情况下，Abaqus 以精确方式达到目标时间；即 Abaqus 根据需要调整时间增量以精确达到目标时间。或者，您可以指示 Abaqus 以近似或宽松方式达到目标时间，仅在目标时间过后才执行协同仿真交换，而 Abaqus 中的时间积分推进不考虑目标时间。在某些情况下，例如当 Abaqus 比 Dymola 需要更多增量时，宽松方法可能更具成本效益。

| **输入文件用法：** | 使用以下选项精确达到目标时间： |
| --- | --- |
| | ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), TIME MARKS=YES (default) ``` 使用以下选项宽松达到目标时间： ``` [*CO-SIMULATION CONTROLS](../key/key-link.md#usb-kws-hcosimulationcontrols), TIME MARKS=NO ``` |

### 准备用于耦合模拟的 Dymola 模型

要为耦合模拟准备 Dymola 模型，您需要：
- 定义命名输入和输出
- 使用 `.DLL` 选项翻译模型以生成 `dymosim.dll`
- 定义协同仿真参数
- 将 Abaqus 中的信号（传感器和执行器）名称映射到 Dymola 中的信号名称

以下各节简要描述这些步骤；这些步骤不旨在描述如何使用 Dymola。有关 Dymola 及其用户界面的更多信息，请参阅相关用户手册。完成这些步骤后，您必须在运行 Dymola 分析的目录中拥有以下文件：
- 与 Dymola 安装对应的 `libdsdll.dll`
- 与所需 Dymola 模型对应的 `dymosim.dll`
- 一个映射（`.sgn`）文件，其内容如下所述

#### 定义命名输入和输出

Dymola 图形编辑器提供了定义输入和输出的最方便方式。从 Modelica 库面板中，将 **Modelica.Blocks.Interfaces.RealInput** 模块拖到画布上以定义输入（蓝色填充三角形）。使用 **Modelica.Blocks.Interfaces.RealOutput** 模块定义输出（空心三角形）。您可以通过弹出对话框更改默认分配的名称。在指定输入和输出模块的名称时，使用所有大写字母非常重要。

#### 使用 `.DLL` 选项翻译模型

在 **Simulation** 面板中，选择 ****Simulation**![](../graphics/images/arrow.gif)**Setup****，然后点击 **Compiler** 选项卡。切换 ****Export model as DLL with API**，然后点击 **OK**。在 **Simulation** 面板中，选择 ****Simulation**![](../graphics/images/arrow.gif)**Translate****。当前工作文件夹中创建了名为 `dymosim.dll` 的文件。您应该特别关注翻译期间发出的警告/错误消息。只有在翻译成功的情况下才能进行成功的协同仿真分析。`dymosim.dll` 文件包含与此特定模型相关的信息。如果需要与其他 Dymola 模型进行协同仿真，请确保更改当前工作目录，以免覆盖 `dymosim.dll` 文件。

#### 定义协同仿真参数

虽然 `dymosim.dll` 文件包含有关 Dymola 模型本身的信息，但它不包含有关分析持续时间、数据交换频率、所需输出频率以及用于 Dymola 时间积分的求解器类型的信息。您需要在单独的文件 `*mapFileName*.sgn` 中使用以下格式提供此信息：

```
Analysis: *analysisTime* *couplingStepSize* *OutputPointsPerCouplingStep* *DymolaSolverType*
```
提供信息的指南如下：- `Analysis` 后面必须紧跟冒号（无空格）。冒号后跟一个空格，然后给出四个数量，用单个空格分隔。
- *analysisTime* 必须与 Abaqus 协同仿真步骤中指定的分析时间匹配。
- *couplingStepSize* 必须与 Abaqus 模型中指定的耦合步骤大小匹配（如果已指定）。设置协同仿真控制的最方便方法是使用默认的 subcycling 方法，并让 Abaqus 建议可变耦合步骤大小。然后您可以将 *couplingStepSize* 指定为等于 *analysisTime*，以允许协同仿真在每个 Abaqus 增量交换数据。
- *OutputPointsPerCouplingStep* 设置 Dymola 中输出生成的频率。值 1 通常就足够了。
- 选择求解器类型时，请参阅 Dymola 用户手册。通常，求解器类型 8 性能最好，尽管比其他求解器略贵。协同仿真分析的计算成本通常由 Abaqus 分析主导，因此 Dymola 求解器的性能对协同仿真分析的性能影响不大。

在大多数情况下，在映射（`.sgn`）文件中指定以下信息就足够了：
```
 Analysis: analysisTime analysisTime 1 8
```

#### 映射信号名称

在实际应用中，Dymola 和 Abaqus 模型很可能由组织中的不同工程师准备，因为有限元分析师可能没有任何控制专业知识，反之亦然。可以合理地假设，Dymola 中用于输入的名称与 Abaqus 传感器和/或的名称不匹配，Dymola 中用于输出的名称与 Abaqus 执行器的名称不匹配。在这种情况下，您可以修改其中一个分析程序中使用的名称。如果不可能，则可以使用映射（`.sgn`）文件来映射要从 Abaqus 交换到 Dymola 的信号名称。您可以在上述 `Analysis` 行之后立即添加以下可选行：

```
Actuators:
*AbaqusActuatorName1   DymolaOutputName1*
*AbaqusActuatorName2   DymolaOutputName2*
*Etc.*
Sensors: 
*AbaqusSensorsName1   DymolaInputName1* 
*AbaqusSensorsName1   DymolaInputName1* 
*Etc.*  
```
提供信息的指南如下：- `Actuators` 后面是一系列名称对，以将 Abaqus 执行器匹配到 Dymola 输出。根据需要使用尽可能多的行（最多 5000）。
- 类似地，`Sensors` 后面是一系列名称对，以将 Abaqus 传感器匹配到 Dymola 输入。根据需要使用尽可能多的行（最多 5000）。

### 执行耦合分析

您可以按 ["Abaqus/Standard、Abaqus/Explicit 和 Abaqus/CFD 执行," 第 3.2.2 节"](pt01ch03s02abx02.md) 中所述执行 Abaqus 作业。您可以按 ["Dymola 模型执行," 第 3.2.5 节"](pt01ch03s02abx05.md) 中所述执行 Dymola 作业。Abaqus 和 Dymola 可以在位于同一网络域的异构和远程平台上运行。出于协同仿真目的，Dymola 必须在 Windows 32 位或 64 位系统上运行，而 Abaqus 可以在任何可用的支持平台上运行。Abaqus 和 Dymola 之间的通信通过传输控制协议（TCP）套接字进行。要在 Abaqus 和 Dymola 之间启动耦合模拟，两个求解器之一必须启动通信过程，而另一个求解器必须连接到已启动的通信过程。Abaqus 或 Dymola 都可以启动通信过程；但是，每个求解器的启动方法不同。以下各节将讨论每种方法。

您需要知道启动通信过程的计算机的网络主机 ID。此 ID 可以是主机名或互联网协议（IP）地址。端口指定允许不同应用维护各自的通信通道。将端口号设置为系统上任何可用的端口。端口号范围为 0 到 65535。通常，1024 以下的端口号由操作系统保留，应避免使用。通过为每个分析分配唯一的端口号，您可以在给定系统上同时运行多个分析作业。

Abaqus 和 Dymola 相关的输入文件不必位于同一目录中。如果两个协同仿真分析在不同的平台上运行且二进制不兼容（例如，Abaqus 作业在 Linux 64 位系统上运行，Dymola 作业在 32 位 Windows 系统上运行），则必须启用 ASCII 数据交换。为此，在用于协同仿真分析的机器的本地 `abaqus_v6.env` 文件中添加以下两行：

```
import os
os.environ['ABAQUS_DCI_FORMAT']='ASCII'
```

#### Abaqus 启动通信过程

要让 Abaqus 启动通信过程，您只需在命令行上提供端口号（而不是主机名）；例如，可以使用以下命令运行 Abaqus 作业：

```
abaqus job=*job_name* port=*portNumber*
```
当 Abaqus 启动通信过程时，Dymola 必须连接到启动 Abaqus 分析的主机和指定端口。Dymola 作业可以使用以下命令：
```
abaqus dymola input=*mapFileName* host=*machineName* port=*portNumber*
```

#### Dymola 启动通信过程

或者，可以使用以下两个命令让 Dymola 启动通信过程：

```
abaqus job=*job_name* host=*machineName* port=*portNumber*
```

```
abaqus dymola input=*mapFileName* port=*portNumber*
```

### 限制

使用 Abaqus 和 Dymola 进行结构对逻辑协同仿真不适用于机械/热系统本身在两个分析程序之间分割的应用。例如，对于轮胎和道路在 Abaqus 中建模而悬架和车身在 Dymola 中建模的耦合分析获得的结果可能会导致不可靠的结果。
